# 🏛️ Architecture & System Design

이 문서는 **Quant Trading Fleet**의 내부 구조와 데이터 흐름에 대해 설명합니다.

## 1. 개요 (Overview)
본 시스템은 **비동기 백엔드(FastAPI)**와 **현대적인 프론트엔드(React)**가 결합된 모놀리식 구조이면서도, 내부적으로는 봇의 실행과 API 서빙이 분리되어 동작하는 하이브리드 설계를 따릅니다.

## 2. 핵심 구성 요소

### A. Bot Manager (비동기 제어 시스템)
- **역할:** 사용자의 요청에 따라 Python 봇 인스턴스를 생성하고, 백그라운드 태스크(`asyncio.Task`)로 관리합니다.
- **기능:** 
    - 봇 상태 관리 (Running, Stopped, Error)
    - Graceful Shutdown (봇 종료 시 열려있는 포지션 안전 청산 시도)
    - 비차단(Non-blocking) 실행: API 서버가 봇의 무한 루프에 영향을 받지 않도록 설계되었습니다.

### B. Persistence Layer (데이터 영속성)
- **SQLite & SQLAlchemy:** 
    - `bot_states`: 각 봇의 설정 및 현재 가동 여부 기록.
    - `trade_history`: 모든 매수/매도 체결 데이터 저장.
    - `app_settings`: 웹 UI에서 변경 가능한 전략 파라미터 저장.
- **파일 기반 저장:** 데이터베이스와 로그 파일은 Docker 볼륨 마운트를 통해 컨테이너 재시작 후에도 유지됩니다.

### C. Broker Abstraction (거래소 추상화)
- **KISBroker:** 한국투자증권 REST API 래퍼. 토큰 캐싱 로직이 내장되어 멀티 봇 환경에서도 API 한도 제한을 방지합니다.
- **BinanceBroker:** CCXT 라이브러리를 활용한 선물 거래소 통신. 주문 정밀도(Precision)와 최소 주문 금액 검증 로직이 포함되어 있습니다.

## 3. 데이터 흐름 (Data Flow)

1. **봇 실행:** 사용자가 웹 UI에서 'Start' 클릭 -> FastAPI 요청 -> `BotManager`가 DB에서 설정을 읽어 백그라운드 태스크 시작.
2. **매매 발생:** 봇 로직에서 '돌파' 감지 -> 브로커 클래스 호출 -> API 주문 전송 -> 성공 시 브로커가 DB의 `trade_history`에 즉시 기록.
3. **상태 모니터링:** React 프론트엔드가 5초마다 API를 폴링하여 실시간 잔고 및 매매 내역 갱신.

## 4. 안정성 설계 (Robustness)

- **타임존 고정:** 모든 컨테이너와 코드는 `Asia/Seoul` 시간대를 사용하도록 강제되어 있어, 장마감 및 예약 매매 로직의 오작동을 방지합니다.
- **중복 매수 방지:** 봇 시작 시 메모리 값에 의존하지 않고, 항상 실제 거래소 API를 통해 현재 보유 수량을 동기화합니다.
- **토큰 공유:** 여러 봇이 동시에 토큰을 요청하여 429 에러가 발생하는 것을 막기 위해 파일 기반 토큰 락(Lock) 및 캐싱을 지원합니다.
